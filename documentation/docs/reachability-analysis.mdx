# Reachability analysis

Reachability analysis is the technique of determining which dependencies in a BOM are actually exercised by your code, and on which paths. A dependency that is declared in a manifest but never called is *present*; a dependency whose functions sit on a data-flow or call-graph path that starts from a real entry point is *reachable*. dep-scan computes reachability before it consults vulnerability data, so a vulnerable package is only escalated when its code is demonstrably on an executed path.

This page is the hub for the reachability documentation set. It states the core ideas and the language-by-analyzer matrix, then links into the concept, language, analyzer, and output chapters where each topic is developed in depth.

## How dep-scan thinks about reachability

dep-scan separates the work into two layers. First, a per-language *slicer* inspects the source (and, for .NET, the assemblies) and emits data-flow and call-graph slices with purl attribution. Second, a shared, purl-keyed engine consumes those slices and decides which package URLs are *reached*. The engine does not care which slicer produced the evidence: an atom slice from a Java project and a golem-derived slice from a Go project are normalized to the same shape before analysis.

Because reachable flows are computed independently of vulnerability databases, dep-scan avoids a common failure mode of reachability tools that rely on a manually maintained database of affected modules. When a CVE lands in the advisory feed, dep-scan simply intersects it with the already-computed reachable set. The concepts behind this design, including the distinction between `direct_purls` and `reached_purls`, are developed in the [reachability model](concepts/reachability-model) chapter.

## The analyzer matrix

Three analyzers are selectable with `--reachability-analyzer`:

| Value                    | What it computes                                                       | When to use it                                                       |
|--------------------------|------------------------------------------------------------------------|----------------------------------------------------------------------|
| `FrameworkReachability`  | Framework-forward reachability from BOM occurrences plus slice purls.  | Default. Fast, language-agnostic, answers "is the package called."   |
| `SemanticReachability`   | Adds endpoint, service, and post-build (binary/container) reachability. | Compliance-grade triage; multi-BOM lifecycle analysis.              |
| `off`                    | Disables reachability; produces a version-based scan only.             | When slicing tooling is unavailable or a quick scan is enough.       |

Every reachability-capable language runs `FrameworkReachability` by default. The two analyzers are compared in depth in the Analyzers category: [Framework reachability](analyzers/framework-reachability) and [Semantic reachability](analyzers/semantic-reachability).

## Language coverage

dep-scan reaches seven language ecosystems through four slicers. The full matrix, including purl types and BOM-generation paths, lives on the [Supported languages](supported-languages) page. In summary:

- atom backs Java/JVM, JavaScript/TypeScript, Python, and PHP. See [JVM, JS, Python, and PHP reachability](languages/jvm-js-python-php-reachability).
- rusi backs Rust. See [Rust reachability](languages/rust-reachability).
- golem backs Go. See [Go reachability](languages/go-reachability).
- dosai backs .NET. See [.NET reachability](languages/dotnet-reachability).

Each language chapter demonstrates a reachable finding and a present-but-unreachable dependency from the hermetic fixture projects under `test/data/`, so you can reproduce the contrast offline.

## Turning reachability on

The documented way to enable reachability is the `research` profile, which configures cdxgen to persist the per-language slice files the engine needs:

```shell
depscan --profile research -t <language> -i <source directory> --reports-dir <reports directory> --explain
```

Reachability is on by default for every supported language once the slices are present, so the explicit `--reachability-analyzer` flag is only needed when you want `SemanticReachability` or `off`. The `--explain` flag adds a human-readable breakdown of the reachable flows to the report.

## From reachability to prioritization and compliance

Reachability feeds two downstream outputs that matter to analysts and compliance teams. First, the prioritization engine tags reachable (and especially endpoint-reachable or malicious-and-reachable) findings with the CycloneDX property `depscan:prioritized` and surfaces them in the Top Priority table; see [How dep-scan prioritizes](concepts/prioritization). Second, the VEX exporter maps reachability to machine-readable status: reachable dependencies become `known_affected`, while present-but-unreachable dependencies become `known_not_affected` carrying the `vulnerable_code_not_in_execute_path` justification. That mapping is the core of the [CSAF VEX guide](output/vex-csaf-guide) and is the strongest argument for investing in reachability when you produce compliance artifacts.
